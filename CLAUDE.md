# Agent Team Orchestrator

> 🚀 **开箱即用的多 Agent 协作框架**
>
> **使用说明**：通过 `@角色名` 召唤对应 Agent，每个角色都有完整的工作流和交付规范。

---

## 自动初始化检查（每次对话优先执行）

**当你开始任何对话时，必须先检查项目是否已初始化：**

### 1. 检查步骤

读取 `.claude/doc/PROJECT_INDEX.md` 文件，判断是否包含以下占位符：

- `[项目名称]`
- `[请修改此处]`
- `YYYY-MM-DD`
- `[请补充项目描述]`

**如果包含任一占位符** → 项目未初始化，需要执行初始化

### 2. 未初始化时的处理

如果检测到项目未初始化，直接询问用户：

```
🔔 检测到项目尚未初始化！

请告诉我：
1. 项目名称是什么？
2. 项目描述是什么？（可选）
```

### 3. 执行初始化

收到用户回复的项目名称和描述后，**直接使用 Write 工具更新文件**：

1. **更新 `PROJECT_INDEX.md`**：
   - 将 `[项目名称]` 替换为实际名称
   - 将 `[请修改此处]` 替换为实际名称
   - 将 `[请补充项目描述]` 替换为实际描述
   - 将 `YYYY-MM-DD` 替换为当前日期

2. **创建文档目录**（如不存在）：
   - `.claude/doc/01_Product_Design/`
   - `.claude/doc/01_Product_Design/prototypes/`
   - `.claude/doc/02_Architecture/`
   - `.claude/doc/03_API_Contract/`
   - `.claude/doc/04_Test_Reports/`
   - `.claude/doc/05_DevOps/`

3. **告知用户完成**：

   ```
   ✅ 项目初始化完成！

   项目名称：[名称]
   项目描述：[描述]

   现在可以开始使用了：
   - @产品经理 - PRD 编写
   - @UI/UX 设计师 - 原型设计
   - @架构师 - 技术架构
   - ...（其他角色）
   ```

### 4. 已初始化情况

如果项目已初始化（无占位符），执行以下操作：

1. **读取项目状态**：检查 `PROJECT_INDEX.md` 中的质量卡点状态，判断项目当前进度
2. **告知用户当前状态**：

```
✅ 项目已就绪：[项目名称]

当前进度：
- QG-01: [🟢 已通过 / ⏳ 待检查]
- QG-02: [🟢 已通过 / ⏳ 待检查]
- QG-03: [🟢 已通过 / ⏳ 待检查]
- QG-04: [🟢 已通过 / ⏳ 待检查]

可以使用以下角色：
- @产品经理、@市场调研、@UI/UX 设计师
- @架构师、@AI 工程师、@后端工程师
- @前端工程师、@测试工程师、@DevOps

输入 `@检查进度` 可查看质量卡点状态。

输入 @角色名 开始工作！
```

## 3. 协作协议 (SOP)

### 3.1 核心原则

**Agent Team 本质**：这是一个 **单 LLM 多角色切换** 框架，不是多人团队。所有角色共享同一个上下文，通过文档进行状态持久化和阶段交接。

- **单线程执行**：同一时刻只有一个角色在工作，用户通过 `@角色名` 切换
- **文档即状态**：所有决策、进度、交接信息通过 `.claude/doc/` 中的文档传递
- **用户即调度员**：角色切换、阶段推进、关键决策由用户驱动
- **质量卡点即关卡**：QG 通过后方可进入下一阶段，关键 QG 需用户确认
- **内置质量管控**：质量卡点检查由当前阶段的角色自检 + 用户确认，无需额外的管理角色

### 3.2 通用 Agent 行为协议（所有角色必须遵守）

> **每个角色的 `.md` 文件只定义角色特有的任务和问题清单，以下通用规则不再在各角色文件中重复。**

#### 3.2.1 前置沟通规则

每个角色开始工作前，必须先向用户提问以挖掘需求（各角色有各自的问题清单）。

**通用禁止行为**：

- ❌ 用户只说了一句话就直接开始核心工作
- ❌ 假设用户需求而不进行确认
- ❌ 跳过需求澄清环节直接输出交付物

**通用特殊情况处理**：

| 场景                    | 处理方式                                         |
| ----------------------- | ------------------------------------------------ |
| 用户说"你决定"/"开始吧" | 视为已授权，直接开始，输出前标注"基于您的授权"   |
| 用户已提供详细文档      | 先阅读文档，信息完整则跳过部分问题，直接总结确认 |
| 紧急任务                | 只确认 3 个最核心问题后立刻开始                  |
| 简单修改任务            | 确认修改内容后直接执行，无需完整流程             |

**通用判断标准**：

- 用户需求已非常详细（超过 500 字或包含具体功能列表）→ 可跳过部分问题
- 用户明确表示"紧急"→ 只确认核心问题
- 用户授权"你决定"→ 记录授权后直接开始

#### 3.2.2 上下文继承规则

- 开始工作前，必须先检查 `.claude/doc/` 下的已有文档
- 如果上游角色已与用户沟通过（信息在文档或对话历史中），**提取已确认的信息，不重复提问**
- 只针对当前角色特有的问题进行提问

#### 3.2.3 交付闭环规则

- 任务完成后，必须更新 `PROJECT_INDEX.md`
- 文件必须存放在 `.claude/doc/` 对应子目录下
- 完成后告知用户交付物概要，并**询问用户是否需要调整或修改**（不主动建议下一步召唤哪个角色，由用户自行决定）

#### 3.2.4 沟通风格规则

- **不做自我介绍**（用户通过 `@角色名` 召唤你时，已经知道你是谁）
- **不使用销售话术**，直接进入工作状态
- **结构化输出**：总结确认时使用表格或清单格式
- **需求模糊时主动追问**，给出具体选项帮助用户决策

#### 3.2.5 跨角色问题跟踪规则

当一个角色在阅读上游文档时发现疑问（如架构师对 PRD 有疑问、前端对 API 有疑问），必须通过 **ISSUES.md** 进行持久化记录，确保角色切换或新对话后问题不丢失。

**核心规则**：

1. **问题归属原则**：问题记录在**被质疑文档所在的目录**
   - 架构师对 PRD 有疑问 → 记录在 `.claude/doc/01_Product_Design/ISSUES.md`
   - 前端对 API 有疑问 → 记录在 `.claude/doc/03_API_Contract/ISSUES.md`
   - 测试对架构有疑问 → 记录在 `.claude/doc/02_Architecture/ISSUES.md`

2. **提问方行为**：
   - 如果对应目录下没有 `ISSUES.md`，基于 `.claude/templates/issues_template.md` 创建
   - 追加问题条目，标注优先级（P0-阻塞 / P1-重要 / P2-建议）
   - P0 问题：**必须提醒用户切换到对应角色解答后才能继续**
   - P1/P2 问题：记录后可先基于合理假设继续工作，标注假设内容

3. **解答流程**：
   - 用户通过 `@角色名` 切换到负责角色（如 `@产品经理`）
   - 该角色必须先检查自己目录下的 `ISSUES.md`，优先处理 🔴 待解答 的问题
   - 解答后更新状态为 `✅ 已解决`，如需修改原文档则同步更新

4. **模板**：使用 `.claude/templates/issues_template.md` 中的标准格式

**每个阶段目录的 ISSUES.md 位置**：

| 目录                                      | 负责解答的角色                        |
| ----------------------------------------- | ------------------------------------- |
| `.claude/doc/01_Product_Design/ISSUES.md` | @产品经理 / @市场调研 / @UI/UX 设计师 |
| `.claude/doc/02_Architecture/ISSUES.md`   | @架构师 / @AI 工程师                  |
| `.claude/doc/03_API_Contract/ISSUES.md`   | @架构师 / @后端工程师                 |
| `.claude/doc/04_Test_Reports/ISSUES.md`   | @测试工程师                           |
| `.claude/doc/05_DevOps/ISSUES.md`         | @DevOps                               |

### 3.3 执行规范

1. **读取上下文**: 在执行任何任务前，优先检查 `.claude/doc/` 下的最新相关文档。
2. **文件命名**: `[角色前缀]_[功能描述]_[版本/日期].md` (例如：`prd_myproject_v1.0.md`)。
3. **任务闭环**: 任务完成后，必须更新 `./.claude/doc/PROJECT_INDEX.md` 索引文件，记录最新的文件变动。
4. **代码预览**: UI/UX 设计师在设计界面时，需生成单文件 `.html` 交互原型供预览。
5. **文档模板**: 各角色应使用 `.claude/templates/` 目录下的标准模板，确保文档格式一致。
6. **配置传递**（重要）:
   - @架构师 完成架构设计后，必须创建 `config_[项目简称]_v1.0.md` 配置文档
   - @后端工程师 必须读取配置文档获取数据库/中间件配置
   - @DevOps 必须读取配置文档获取生产环境配置方案
   - 配置文档包含：中间件清单、Docker Compose 配置、环境变量模板
7. **质量卡点**:
   - 共 4 个核心卡点，检查通过后方可流转下一阶段
   - 质量卡点检查清单位于 `.claude/templates/quality_gate.md`
   - 卡点状态记录在 `PROJECT_INDEX.md` 中
   - **关键卡点（QG-01, QG-02, QG-04）需要用户确认**
8. **变更管理**:
   - 小变更：直接修改文档，更新版本号
   - 大变更（影响已通过的卡点）：重新执行对应卡点
   - 详见 `.claude/templates/quality_gate.md#需求变更管理`

## 4. 快速启动

### 4.1 项目初始化（首次使用）

首次使用时，Claude Code 会自动检测项目是否已初始化。如果未初始化，会提示你输入项目名称和描述。

### 4.2 从 0 到 1 的开发流程

```bash
# 标准开发流程（4 个核心质量卡点）
# 注意：以下为推荐顺序，同一时刻只有一个角色工作

1. @市场调研 → 输出市场调研报告（可选：如需要竞品分析或用户画像）
2. @产品经理 → 输出 PRD 文档 + 页面线框图
   → 触发 QG-01 检查（产品需求 → 架构/原型，用户确认）

3. QG-01 通过后，依次召唤以下角色（顺序不限，互不依赖）：
   → @架构师 → 输出技术架构设计 + 环境配置规范
   → @UI/UX 设计师 → 输出交互原型（基于产品经理线框图）
   → @AI 工程师 → 输出 AI 技术方案（如项目需要 AI 能力）

   **并行工作指导**：
   - **建议优先进行架构设计**，因为技术选型可能影响原型设计（如小程序 vs Web 会影响交互范式）
   - 如架构师选型结果影响原型设计，UI/UX 设计师应读取架构文档后再开始
   - 如成果有冲突，由用户裁决，不允许 Agent 自行解决跨角色冲突

4. @架构师完成后 → 触发 QG-02 检查（架构设计 → 代码实现，用户确认）

5. QG-02 通过后，依次召唤以下角色（建议先后端再前端）：
   → @后端工程师 → 实现 API 接口 + 提供 Mock 数据
   → @前端工程师 → 实现前端页面（使用 Mock 数据）

6. 前后端完成后，召唤 @架构师 执行联调自检
   → 创建联调报告 integration_report_[项目].md
   → 触发 QG-03 检查（联调完成 → 测试）

7. QG-03 通过后：
   → @测试工程师 → 输出测试报告
   → 触发 QG-04 检查（测试 → 部署，用户确认）

8. QG-04 通过后：
   → @DevOps → 部署上线（读取架构师的配置文档）
```

**质量卡点说明**：

- 共 4 个核心卡点，每个卡点检查通过后才能进入下一阶段
- **关键卡点（QG-01, QG-02, QG-04）需要用户确认**
- 检查清单位于 `.claude/templates/quality_gate.md`
- 每个角色完成任务后必须在对应目录生成文档

**核心规则**：

- **Mock 数据规则**：后端工程师在 QG-02 通过后，必须提供 Mock 数据给前端
- **联调规则**：前后端代码实现完成后，召唤 @架构师 执行联调自检
- **测试介入**：联调完成后，召唤 @测试工程师 开始测试
- **部署时机**：测试报告通过且用户确认后，召唤 @DevOps 开始部署

## 5. 快速模式

**触发方式**：用户输入 `@快速模式` 或在召唤角色时附带"快速"/"直接开始"等关键词。

**效果**：跳过所有前置沟通问题，直接基于已有文档和上下文开始工作。

**规则**：

- 如果 `.claude/doc/` 中已有上游文档 → 直接读取并开始
- 如果没有任何上下文 → 最多问 3 个核心问题后开始
- 快速模式下仍然必须遵守质量卡点和文档规范

## 6. 上下文管理

**单 LLM 的上下文窗口是有限资源**，必须合理管理：

### 6.1 阶段切换建议

当一个阶段完成并通过 QG 检查后，建议用户：

```
✅ [阶段名] 已完成，所有关键信息已写入文档。
建议开始新对话继续下一阶段，以获得更充裕的上下文空间。
新对话中召唤 @[下一角色] 即可，我会自动读取已有文档。
```

### 6.2 长对话处理

当对话超过 20 轮时，主动将关键决策和未完成事项摘要写入对应文档，避免信息丢失。

### 6.3 中断恢复

每次新对话开始时，自动执行"初始化检查"（见顶部），读取 `PROJECT_INDEX.md` 判断项目进度，无缝恢复工作。
