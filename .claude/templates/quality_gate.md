## 使用说明

**质量卡点是什么？**

质量卡点是阶段之间的"关卡"，确保上游产出物质量合格后才允许流转到下游。

**核心原则**：
1. **用户评审**：每个卡点必须经过 **@用户** 评审确认后才能通过
2. **Agent 自检**：下游 Agent 在开始工作前，必须自检上游卡点是否通过
3. **状态记录**：卡点状态必须记录在 `PROJECT_INDEX.md` 中，供所有 Agent 查看

**如何使用？**
1. 完成任务后，对照检查清单逐项确认
2. **邀请 @用户 评审**，用户确认通过后标记为"通过"
3. 在 `PROJECT_INDEX.md` 中记录卡点通过时间和结果
4. 下游 Agent 开始前，必须检查 `PROJECT_INDEX.md` 确认上游卡点已通过

## 异常处理流程

### 当上游文档质量不达标时

**场景**：@产品经理 交付的 PRD 质量不佳，@架构师 无法开始工作

**处理流程**：
1. **发现方**（如@架构师）列出 PRD 中的模糊/缺失点
2. **反馈给 @项目经理**，由 @项目经理 协调 @产品经理 补充
3. **不直接进入下一阶段**，避免问题放大到下游
4. **记录到风险日志**：`.claude/doc/00_Project_Management/risk_log_[项目].md`

```mermaid
graph LR
    A[发现上游文档问题] --> B[列出问题清单]
    B --> C[反馈给@项目经理]
    C --> D[@产品经理 补充/修正]
    D --> E[重新提交检查]
    E --> F{检查通过？}
    F -->|是 | G[进入下一阶段]
    F -->|否 | D
```

### 当质量卡点检查不通过时

**场景**：QG 检查发现有未通过项

**处理流程**：
1. **检查者**记录未通过项到 `PROJECT_INDEX.md` 质量卡点状态表
2. **P0 问题**（阻塞性问题）：立即修复，修复后重新检查
3. **P1/P2 问题**（非阻塞性）：
   - 可记录为技术债务
   - 在下一个迭代中修复
   - 需 @项目经理 评估风险后决定是否放行
4. **升级机制**：
   - 连续 2 次检查不通过 → 通知 @项目经理
   - 3 次不通过 → 上报项目风险，考虑调整计划

### 当需求不明确时

**场景**：用户输入模糊，无法确定具体需求

**处理流程**：
1. **接收方**先召唤对应角色 Agent（如@产品经理）进行需求澄清
2. **使用 AskUserQuestion 工具**向用户提问，获取明确信息
3. **更新 PRD 或项目计划**，记录澄清后的需求
4. **重新通过质量卡点检查**
