## 使用说明

**质量卡点是什么？**

质量卡点是阶段之间的"关卡"，确保上游产出物质量合格后才允许流转到下游。

**核心原则**：

1. **用户确认**：关键卡点由 **@用户** 确认通过
2. **Agent 自检**：下游 Agent 在开始工作前，必须自检上游卡点是否通过
3. **状态记录**：卡点状态必须记录在 `PROJECT_INDEX.md` 中

**如何使用？**

1. 完成任务后，对照检查清单逐项确认
2. 关键卡点邀请 @用户 确认
3. 在 `PROJECT_INDEX.md` 中记录卡点通过状态

## 用户确认交互规范

**关键卡点（QG-01, QG-02, QG-04）必须使用以下格式请求用户确认：**

```
📋 **质量卡点 [QG-XX] 检查报告**

**阶段**：[XX] → [XX]
**检查者**：@[角色名]

**检查结果**：
- ✅ [检查项 1 - 通过原因]
- ✅ [检查项 2 - 通过原因]
- ...

**交付物清单**：
- 📄 [文档名称 + 存放路径]

**请您确认**：
- ✅ 确认通过：回复“确认”或“通过”
- ❌ 需要修改：回复具体修改意见
- ❓ 有疑问：提出您的问题
```

**Agent 处理规则**：

- 用户回复“确认/通过/OK/可以”→ 标记卡点通过，更新 `PROJECT_INDEX.md`
- 用户回复修改意见 → 执行修改后重新提交确认
- 用户未回复 → **不得自行跳过**，必须等待或提醒用户

## QG-01: 产品需求 → 架构/原型

**触发时机**：@产品经理 完成 PRD + 线框图后，@架构师 开始技术选型、@UI/UX 开始原型设计前

### 检查清单

- [ ] **功能清单完整**：P0/P1/P2 优先级明确，无遗漏
- [ ] **验收标准明确**：每个 P0/P1 功能至少有 2 条 AC
- [ ] **用户故事规范**：格式正确（作为...我想要...以便...）
- [ ] **非功能需求量化**：性能、安全、兼容性有具体指标
- [ ] **业务流程图完整**：核心流程有 Mermaid 图或 User Flow
- [ ] **页面线框图完整**：核心页面有低保真线框图
- [ ] **用户确认通过**：@用户 已确认需求符合预期

### 检查流程

```
PRD + 线框图完成 → 用户确认 → 自检 + 文档检查 → 所有检查项通过 → 标记 QG-01 通过
```

**流转说明**：

- QG-01 通过后，@架构师 开始架构设计，@UI/UX 设计师 开始交互原型设计

## 需求变更管理

**场景**：在开发过程中，用户提出需求变更或新增功能

**处理流程**：

1. **评估变更影响范围**：
   - 如仅影响代码实现 → @后端/@前端 直接修改，更新相关文档
   - 如影响 PRD/原型 → 更新 PRD，重新通过 QG-01
   - 如影响架构 → 更新架构设计，重新通过 QG-02

2. **文档更新**：
   - 更新受影响的文档版本号和变更记录
   - 在 `PROJECT_INDEX.md` 中记录变更说明并更新状态

**简单规则**：

- 小变更：直接修改文档，更新版本号
- 大变更（影响已通过的卡点）：重新执行对应卡点

## 异常处理

### 当上游文档质量不达标时

**场景**：上游交付物质量不佳，下游无法开始工作

**处理流程**：

1. 发现方列出模糊/缺失点
2. 反馈给上游角色补充
3. 不直接进入下一阶段
4. 记录到风险日志

### 当质量卡点检查不通过时

**处理流程**：

1. 检查者记录未通过项到 `PROJECT_INDEX.md`
2. P0 问题（阻塞性问题）：立即修复，修复后重新检查
3. P1/P2 问题（非阻塞性）：可记录为技术债务，后续修复
4. 修复完成后，由原检查者重新执行检查清单
5. 重新检查通过后，更新卡点状态为 🟢 已通过
